# OTD 區域計算器 (OTD Area Calculator) v1.0 技術總結報告

這是一份關於「OTD 區域計算器 v1.0」的完整開發與技術架構總結。
本軟體旨在解決 osu! 玩家在設定 OpenTabletDriver (OTD) 繪圖板打擊範圍時，因為手感偏差而反覆盲調的痛點。透過追蹤玩家在實際遊玩時的真實下筆軌跡，運用資料科學演算法剔除極端雜訊，最終反推出最適合該玩家的**「最佳鏡設區域 (Sweet Spot)」**絕對物理座標，讓玩家能一鍵無痛複製回 OTD 控制面板中。

---

## 1. 核心架構與設計哲學

我們在一開始遭遇了 `v2` 到 `v4` 版本間許多硬體與多執行緒卡死的挑戰，為此 `v1.0` 進行了「從底層驅動到上層 UI」的徹底重構：

*   **框架選擇**：採用 Python 3.13 作為核心，搭配 **PySide6 (Qt6)** 建構現代化的非同步圖形介面。不僅達成了全黑的電競質感 (Dark Mode)，還確保了高幀率的「流動即時畫布」。
*   **硬體通訊**：捨棄了不穩定的 `win32api` 或 `TabletEvent` 掛鉤，改為使用底層的 **`hidapi (hid)`** 直接以非阻塞 (Non-blocking) 模式嗅探繪圖板的 USB/藍牙封包。
*   **非同步架構**：為了應付現代繪圖板與電競滑鼠高達 1000Hz (每秒 1000 次) 的資料回報率，我們將硬體輪詢 (Hardware Polling) 邏輯從 UI 主執行緒徹底剝離，透過繼承 `QThread` 撰寫了專屬的 `HIDPollingThread`。這確保了主介面絕對不會因為等待 IO 而發生凍結 (Freeze)，達成了 **0 掉封包 (0 Packet Drops)** 的流暢記錄。

---

## 2. 解決的關鍵技術瓶頸

開發過程中最艱困的莫過於解決**「X 軸嚴重壓縮，畫出來的圓形變成垂直抖動線」**的魔王級 Bug。這涉及到對硬體底層封包的逆向工程與除錯：

### A. Wacom 24-bit 隱藏對齊格式 (The 24-bit Aligned Anomaly)
舊版程式碼(或大多數網路上的開源腳本) 普遍假設 HID 的封包結構是連續的 16-bit：
*   `X = data[2] | data[3]<<8`
*   `Y = data[4] | data[5]<<8`

然而，透過我們在背景植入的微型封包側錄器 (Packet Sniffer) 擷取使用者的真實操作日誌後，我們破解了 Wacom Intuos S (CTL-4100) 在拔除原廠驅動後的**裸機傳輸真相**：它實際上是採用了罕見的 **對齊式 24-bit (24-bit Aligned)** 規格：
*   `X = data[2] | data[3]<<8 | data[4]<<16`
*   `Y = data[5] | data[6]<<8 | data[7]<<16`

舊版的解析器因為從第 `[4]` 個位元組就開始讀取 Y 軸，導致它把屬於 X 軸的高位元空白填充 (Padding `00`) 跟 Y 軸黏在一起解讀了。這使得 Y 軸的世界常數被無限往上衝撞，最後 X 軸在畫面長寬比中被相對嚴重壓縮。修正為 24-bit 讀取後，座標比例瞬間獲得了完美釋放。

### B. 驅動斷線的 0xFF00 填充陷阱 (The 65280 Overflow Bug)
測試中我們觀察到 Y 軸偶爾還是會暴走至 `65280`。將其轉換為十六進制後發現是 `0xFF00`。
這是因為當繪圖筆離開板子感應高度時，硬體會發送 `0xFF00` 作為「斷訊填充 (Padding)」。如果程式單純把它當成無號整數 (Unsigned) 處理，就會變成 `65280` 的超大座標；如果當成有號整數 (Signed 16-bit) 處理，其實代表 `-256`。
我們在資料輸入層加入了嚴格的防護牆：**直接過濾任何小於 0 或大於 40000 的物理極限謬誤座標**，這徹底讓懸空亂飄的干擾點絕跡。

### C. 設備識別衝突 (The Generic Razer Interception)
為了支援非 Wacom 品牌的雜牌手寫板，程式一度寫入了「 Generic 0x0D Digitizer (通用數位板通道)」的抓取機制。孰料，這導致了使用者的 Razer Viper V3 Pro 電競滑鼠裡面的隱藏數位通道優先劫持了軟體，導致繪圖板連線失敗。
最終方案確立為：**實施嚴格的 Wacom Vendor ID (0x056A) 與指定 Product ID (如 0x0374, 0x037A 等) 白名單**，精準鎖定目標設備。

## 3. 資料科學演算法：精準切割甜蜜點

當完美的原始座標被捕捉後，如何從數萬個雜亂的軌跡點中，框選出真正代表玩家的「最佳鏡設區域」，靠的是資料處理演算法的升級：

### 1. 淘汰粗暴的百分比剪裁法 (The 2% Cut Failure)
舊版 `v2` 採用了極度生硬的做法：直接把所有座標點依照大小排序，然後無腦切掉最前面的 2% 和最後面的 2%。這導致了一個嚴重缺陷：**如果玩家當天只畫了很小的圈圈，正常的邊界點也會被當成雜訊切掉，結果算出來的範圍越變越瞎。**

### 2. 引進四分位距 (IQR, Interquartile Range) 異常檢測
在 `v1.0` 中，我們導入了統計學與機器學習前處理中常見的 **IQR 演算法**：
1. 取出所有 X (或 Y) 座標的 第 25 百分位數 ($Q_1$) 與 第 75 百分位數 ($Q_3$)。
2. 計算兩者差距即為 IQR ($IQR = Q_3 - Q_1$)。
3. 嚴格定義：只有小於 $Q_1 - 1.5 \times IQR$ 或大於 $Q_3 + 1.5 \times IQR$ 的超級極端值，才會被系統認定為真正的「瞬移雜訊 (Cursor Teleport Glitches)」。
這樣一來，無論玩家畫大畫小，正常的極限操作邊界都會被完美保留，只有偶發的通訊電流干擾會被精準剔除。

### 3. 動態邊界自適應 (Dynamic Auto-Calibrating Boundaries)
在不知道目前使用者手繪板型號最大極限寬高的情況下，我們捨棄了去資料庫死背 `max_x=15200` 這類硬參數的執念。程式現在擁有自動學習能力：
它會以預設的官方帳面數據 (`15200x9500`) 開局，但只要程式持續開著，任何突破記錄的封包座標進來，它都會動態擴張這個世界邊界 `dyn_max`，實現 100% 符合當前系統驅動狀態的完美比例映射。

### 4. 絕對座標反演 (OTD Absolute Coordinate Mapping)
透過極大化與極小化的有效座標點，應用程式會算出玩家專屬的活動範圍框。
為了完全相容 OTD 的要求格式，系統會計算出該活動框的正中心點距離繪圖板左上角 `(0,0)` 絕對距離，並按照 `[比例] x [物理總長度(mm)]` 轉換成準確到小數點後二位的 **偏移值 (Offset X / Y)**，供玩家直接在 OTD 輸入套用。

## 4. 結論
透過結合**底層二進位協定逆向**、**非同步 IO 系統設計**以及**資料科學統計除雜訊**，OTD 區域計算器從一個不穩定的雛形，正式蛻變成為 osu! 社群中真正具備實戰價值、穩定且能完美跨越硬體型號差異的電競級輔助工具。
